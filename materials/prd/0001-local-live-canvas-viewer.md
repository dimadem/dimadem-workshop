# Локальный live-вьюер канваса

## Постановка проблемы

Артист пишет Python-кисти, которые подключаются к shared-серверу воркшопа и шлют штрихи через WebSocket. Чтобы увидеть результат, сегодня нужно открыть серверный вьюер `/canvas/view`, который показывает **накопленную картину** всех артистов на shared-канвасе. Это полезно для финального просмотра, но плохо подходит для наблюдения за процессом: невозможно вычленить активность отдельных артистов в моменте — глаз тонет в накопленных штрихах десятков соседей.

Артисту нужен локальный инструмент, который показывает только то, что приходит по WS **прямо сейчас**, как пульс shared-канваса.

## Решение

Локальный standalone HTML-файл `canvas/canvas.html`, который:

- подключается к main-канвасу (`ws://195.133.25.57/canvas/ws`),
- работает как **live-monitor**, а не как viewer накопленной картины: каждые 1/30 секунды холст полностью очищается и перерисовывается только тем, что пришло за последний кадр,
- даёт возможность отфильтровать слои по `artist_name` через dropdown, чтобы видеть пульс конкретных артистов.

Файл открывается двойным кликом в браузере — никакой сборки, никаких зависимостей, никакого сервера для самого HTML.

## User Stories

1. Как артист, я хочу открыть `canvas/canvas.html` двойным кликом, чтобы сразу увидеть живой поток штрихов с shared-канваса без локального бэкенда или сборки.
2. Как артист, я хочу, чтобы вьюер автоматически подключался к main-канвасу `ws://195.133.25.57/canvas/ws` при загрузке страницы, чтобы не настраивать соединение вручную.
3. Как артист, я хочу видеть штрихи всех артистов в реальном времени, чтобы наблюдать за активностью на shared-канвасе.
4. Как артист, я хочу, чтобы холст в браузере полностью помещался в окно с сохранением пропорций, чтобы видеть всю область независимо от размера экрана.
5. Как артист, я хочу, чтобы каждый штрих жил на экране ровно один кадр (1/30 сек), чтобы получить эффект анимации «пульса» вместо накопленной картины.
6. Как артист, я хочу видеть кнопку фильтра в верхней части страницы, чтобы быстро открыть список активных артистов.
7. Как артист, я хочу, чтобы по клику на кнопку фильтра разворачивался dropdown с чекбоксами по одному на каждого артиста, штрихи которого приходили по WS.
8. Как артист, я хочу, чтобы новый артист, чьи штрихи только что появились, автоматически добавлялся в dropdown с включённой галочкой, чтобы я сразу видел его активность.
9. Как артист, я хочу снять галочку у любого артиста, чтобы исключить его штрихи из текущего и последующих кадров без отключения от сервера.
10. Как артист, я хочу, чтобы при получении сервером команды `clear` мой локальный список артистов обнулялся и dropdown становился пустым, отражая полный сброс состояния shared-канваса.
11. Как артист, я хочу, чтобы при разрыве WS-соединения вьюер просто переставал получать новые штрихи без ошибочных диалогов и попыток переподключения — для перезапуска я обновлю страницу сам.
12. Как артист, я хочу, чтобы snapshot, приходящий при подключении, обрабатывался по тем же правилам, что и delta (вспышка и забыть) — никаких отдельных режимов для «старта» и «работы».

## Решения по реализации

**Артефакт**: один файл `canvas/canvas.html`. Никаких отдельных JS/CSS файлов, никакого сборщика. Inline `<style>` и `<script>`.

**Конфигурация**: две JS-константы в начале файла фиксируют адрес сервера:

```js
const SERVER_HTTP = "http://195.133.25.57";
const SERVER_WS   = "ws://195.133.25.57/canvas/ws";
```

Переменности по другим канвасам нет — `*-test` использует только Python-тест (`tests/server/test_canvas.py`), вьюер работает только с main.

**Размер холста**: нативные `width`/`height` атрибуты `<canvas>` берутся из поля `canvas` сообщения `open` (это первое сообщение WS, оно содержит `{width, height}`). Дополнительный HTTP-запрос к `/canvas/config` не нужен — данные те же, источник единый. CSS-стилем `max-width: 100vw; max-height: 100vh; object-fit: contain` (или эквивалент через `width: auto; height: auto; max-*: 100%`) холст вписывается в окно с сохранением пропорций.

**WS-протокол** — следуем контракту из CLAUDE.md и проверенному в `tests/server/test_canvas.py`:

- При получении `open` — берём `width/height`, выставляем размер `<canvas>`.
- При получении `snapshot` — заливаем `strokes` в буфер текущего кадра (как обычный delta).
- При получении `delta` — заливаем `strokes` в буфер текущего кадра.
- При получении `clear` — сбрасываем буфер текущего кадра, список артистов, состояние dropdown.
- Любое неизвестное сообщение игнорируем.
- При закрытии WS — ничего не делаем (вьюер «замирает», артист обновит F5).

**Модули** (логические, внутри одного файла):

1. **Модуль соединения (`connection`)**. Открывает WS, парсит JSON, эмитит события `open`, `strokes` (snapshot+delta), `clear`. Не знает ни про canvas, ни про фильтр. Глубокий: интерфейс из 3 событий, внутри парсинг + lifecycle WS.
2. **Модуль рендера (`renderer`)**. Держит ссылку на `<canvas>` и буфер текущего кадра. На внешний сигнал `addStrokes(list)` дописывает в буфер. Внутри — `requestAnimationFrame` loop, throttled до 30fps: на тик `ctx.clearRect()` → отрисовать строки буфера, у которых `artist_name` сейчас включён в фильтре → очистить буфер. Внешний сигнал `reset()` обнуляет буфер.
3. **Модуль фильтра (`artistFilter`)**. Держит `Map<artist_name, enabled: bool>`. Принимает `observe(artist_name)` (видели штрих) → добавляет нового с `enabled=true`. Принимает `isEnabled(artist_name)` (renderer спрашивает). Принимает `reset()` (на `clear`). Эмитит изменение состояния → UI dropdown синхронизируется.
4. **Модуль UI (`dropdown`)**. Кнопка-триггер сверху + раскрывающийся список с чекбоксами. Подписан на изменения фильтра, рендерит чекбоксы. Клик по чекбоксу → переключает состояние в `artistFilter`.

Эти модули логические, не файловые. Между ними — небольшие функции/объекты в одном `<script>`.

**Цикл данных одного штриха**:
`WS message` → `connection` парсит → `strokes` event → `artistFilter.observe(artist_name)` + `renderer.addStrokes(...)` → следующий тик 30fps → `renderer` отрисовывает только включённые → стирает буфер.

**Стиль штриха**: линия от `(x1,y1)` до `(x2,y2)`, толщина `w`, цвет `color`. `lineCap='round'`, `lineJoin='round'` — стандарт для штрихов кистью, минимизирует артефакты на стыках.

**Фон холста**: белый (`#ffffff`) — соответствует фактическому использованию в воркшопе (артисты часто заливают белым фон отдельным штрихом, см. `tests/server/test_canvas.py`).

**Расположение в репозитории**: файл `canvas/canvas.html` — путь зафиксирован в `CLAUDE.md`.

## Решения по тестированию

**Хороший тест проверяет внешнее поведение, а не детали реализации.** Для live-вьюера «внешнее» — это пиксели на холсте: пришёл штрих по WS → ожидаемые пиксели окрасились в кадре. Всё остальное (внутренние модули, имена переменных, структура буфера) — детали, не тестируем.

**Инструмент**: Playwright MCP-плагин — управляет браузером, открывает страницу, делает screenshot, читает DOM, может выполнять JS на странице. Тест прогоняется руками или из агента; CI-инфраструктуры для браузерных тестов не заводим.

**Сценарий E2E-теста**:

1. Открыть `canvas/canvas.html` в Playwright (`browser_navigate`). Файл подаётся либо через `file://`, либо через локальный `python -m http.server` — что заработает с WS-соединением.
2. Дождаться `open` от сервера (через `browser_wait_for` на состояние «canvas с ненулевыми размерами» — или через `browser_evaluate` на готовность WS).
3. Из теста зарегистрировать уникального артиста (`POST /canvas/register` с именем вида `viewer<random>`) и параллельно открыть второй WS-сокет в роли «бродкастера» — точно тот же приём, что в `tests/server/test_canvas.py`.
4. Послать заведомо контрастный штрих (например, толстую чёрную линию на белом фоне в известных координатах).
5. **Сразу** после отправки (до того как renderer стер кадр) — `browser_take_screenshot` нужной области canvas. Live-режим даёт штриху ровно один кадр жизни — тайминг критичен; в реализации теста придётся либо синхронизироваться через `requestAnimationFrame`, либо временно затормозить render-loop через `browser_evaluate`.
6. Проверить через `browser_evaluate` (`canvas.getContext('2d').getImageData(...)`), что в ожидаемом месте есть не-белые пиксели.

**Что покрывает этот тест**:

- Подключение к WS работает.
- Сообщения парсятся.
- Координаты штриха корректно мапятся на пиксели canvas.
- 30fps-цикл успевает отрисовать штрих до своей же очистки.

**Что НЕ покрывается этим тестом** (сознательно):

- UI-фильтр (dropdown с чекбоксами) — отдельный сценарий: послать штрихи от двух артистов, через `browser_click` снять галочку у одного, послать новые штрихи, проверить что только один артист отрисован.
- `clear` — отдельный сценарий: послать штрихи, дёрнуть `POST /canvas/clear`, проверить что dropdown пустой через `browser_evaluate`/`browser_snapshot`.

**Где живёт тест**: новый файл в `tests/server/` (например `test_viewer.py`), в логике повторяющий `test_canvas.py` (тот же `conftest.py`, тот же приём с уникальными `artist_name`), но дополнительно использующий Playwright MCP. Тест работает с **main**-канвасом (вьюер подключается только к main — см. PRD); коллизия с другими прогонами исключена уникальным именем артиста.

Серверный контракт уже покрыт `tests/server/test_canvas.py` — этот тест опирается на тот же фундамент и проверяет именно клиентскую отрисовку.

## За рамками

- Накопительный режим, в котором штрихи остаются на холсте.
- Зум, пан, экспорт картинки в PNG.
- Реконнект WS, индикатор статуса соединения.
- Подключение к `/canvas/ws-test` (тестовый канвас обслуживается только Python-тестом).
- UI-конфигурация адреса сервера (поле ввода, query-параметр и т.п.).
- Сохранение состояния фильтра между перезагрузками страницы.
- Отрисовка нелинейных примитивов (текст, фигуры) — контракт сервера определяет только линейный stroke.
- CI-инфраструктура для Playwright-тестов: тесты запускаются руками/из агента, в pipeline не интегрируем.

## Дополнительные заметки

- Один файл, никакой сборки — пользователь должен иметь возможность открыть `canvas.html` через `file://` без локального сервера.
- В CLAUDE.md уже зафиксирован путь `canvas/canvas.html` как standalone HTML-вьюер; этот PRD реализует его впервые (папка `canvas/` сейчас пустая).
- Live-режим — сознательное архитектурное решение, не упрощение по нехватке времени. Это другой продукт по сравнению с серверным `/canvas/view`: тот показывает картину, этот показывает пульс.
