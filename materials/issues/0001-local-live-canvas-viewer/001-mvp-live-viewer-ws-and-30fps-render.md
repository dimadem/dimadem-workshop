## Что строим

Standalone HTML-файл `canvas/canvas.html`, который артист открывает двойным кликом и сразу видит живой пульс shared-канваса. Никакого локального сервера, никакой сборки.

Сценарий end-to-end:

1. Артист открывает `canvas/canvas.html` через `file://` (или локальный `python -m http.server` — что заработает с WS).
2. Файл при загрузке автоматически открывает WS на `ws://195.133.25.57/canvas/ws`.
3. Из первого сообщения `open` берёт `canvas.width` / `canvas.height` и выставляет нативные атрибуты `<canvas>`. CSS вписывает холст в окно с сохранением пропорций (`max-width: 100vw; max-height: 100vh; object-fit: contain` или эквивалент).
4. Все приходящие штрихи (`snapshot` и `delta` обрабатываются единообразно) кладутся в буфер текущего кадра.
5. 30fps-loop через `requestAnimationFrame`: на тик `clearRect()` → отрисовать штрихи буфера → очистить буфер. Каждый штрих живёт ровно один кадр (≈33 мс).
6. Стиль штриха: линия от `(x1,y1)` до `(x2,y2)`, толщина `w`, цвет `color`, `lineCap='round'`, `lineJoin='round'`.
7. Фон холста: `#ffffff` (белый).
8. При разрыве WS — вьюер просто перестаёт получать штрихи; никаких диалогов и автореконнекта.
9. Неизвестные типы WS-сообщений игнорируются.

Артефакт — **один файл** `canvas/canvas.html` с inline `<style>` и `<script>`. Адрес сервера зафиксирован двумя JS-константами `SERVER_HTTP` и `SERVER_WS` в начале файла.

Фильтр по `artist_name` и обработка `clear` в этой задаче **не реализуются** — на первом этапе вьюер просто рисует всё, что приходит. Это нужно, чтобы получить рабочий tracer bullet через все слои (WS → парсинг → буфер → 30fps render → пиксели) и потом наращивать фильтр и сброс поверх.

## Критерии выполненной задачи

- [x] Существует файл `canvas/canvas.html` — единственный артефакт, без внешних JS/CSS.
- [x] При открытии через `file://` или локальный HTTP-сервер вьюер автоматически подключается к `ws://195.133.25.57/canvas/ws`.
- [x] Размер `<canvas>` берётся из поля `canvas` сообщения `open`; дополнительный запрос к `/canvas/config` не выполняется.
- [x] Холст вписывается в окно с сохранением пропорций при любом размере экрана.
- [x] Сообщения `snapshot` и `delta` обрабатываются одинаково — штрихи попадают в буфер текущего кадра.
- [x] 30fps-loop очищает холст и перерисовывает только содержимое буфера; каждый штрих живёт ровно один кадр.
- [x] При закрытии WS вьюер «замирает» без ошибочных диалогов и без попыток реконнекта.
- [x] Неизвестные типы WS-сообщений игнорируются без падений.
- [x] Тесты на клиентскую отрисовку согласно «Решения по тестированию» родительского PRD: Playwright-сценарий с уникальным `artist_name`, регистрацией через `POST /canvas/register`, отправкой контрастного штриха через второй WS-сокет, проверкой не-белых пикселей на `<canvas>` через `browser_evaluate` + `getImageData`. Тест живёт в `tests/server/` (например, `test_viewer.py`), повторяет паттерн `test_canvas.py` (уникальное имя через `new_artist_name`, `conftest.py`).
