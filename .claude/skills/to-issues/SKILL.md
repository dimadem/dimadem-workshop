# To Issues

Разбить план на независимо выполнимые задачи через вертикальные задачи (tracer bullets).

## Процесс

### 1. Собери контекст

Работай с тем, что уже есть в контексте диалога. Если пользователь передал путь к документу как аргумент — прочитай его целиком.

### 2. Изучи кодовую базу (опционально)

Если ты ещё не изучил кодовую базу — сделай это, чтобы понять текущее состояние кода. Заголовки и описания issue должны использовать словарь доменной области проекта и учитывать ADR (architectural decision records) в рабочей части кода.

### 3. Проектируй вертикальные задачи

Разбей план на задачи-**tracer bullets**. Каждая задача — это тонкий вертикальный путь через ВСЕ слои интеграции приложения, а НЕ горизонтальный пласт одного слоя репозитория.

Сверься с разделом «Решения по тестированию» PRD: если задача затрагивает модуль, который согласно PRD должен покрываться тестами, в критериях выполнения этой задачи обязательно укажи отдельный пункт про написание тестов (см. шаблон ниже). Если модуль в PRD как тестируемый не значится — пункт про тесты не добавляй, не плоди пустые галочки.

Задачи бывают 'HITL' или 'AFK'. HITL-задачи требуют участия человека — например, архитектурного решения или ревью дизайна. AFK-задачи можно реализовать без участия человека. По возможности предпочитай AFK поверх HITL.

<vertical-slice-rules>
- Каждая задача доставляет изолированный, но ПОЛНЫЙ путь через все слои (схема, API, UI, тесты)
- Собранную задачу можно продемонстрировать или верифицировать самостоятельно
- Предпочитай много изолированных компактных задач, малому количеству больших размытых задач.
</vertical-slice-rules>

### 4. Согласуй с пользователем

Представь предлагаемую разбивку как нумерованный список. Для каждой задачи покажи:

- **Заголовок**: короткое описательное название
- **Тип**: HITL / AFK
- **Блокируется**: какие другие задачи (если есть) должны быть завершены раньше
- **Покрываемые user stories**: какие пользовательские истории закрывает задача (если в исходных материалах они есть)

Спроси пользователя:

- Подходит ли гранулярность? (слишком крупно / слишком мелко)
- Корректны ли зависимости между задачами?
- Нужно ли какие-то задачи объединить или раздробить дальше?
- Правильно ли расставлены метки HITL и AFK?

Итерируй, пока пользователь не утвердит разбивку.

### 5. Запиши issue в файлы

Для каждой утверждённой задачи создай отдельный markdown-файл по пути:

```
materials/issues/<имя-папки-из-имени-PRD-файла>/<NNN>-<kebab-slug>.md
```

- `<имя-папки-из-имени-PRD-файла>` — имя PRD-файла без расширения. Например, для `materials/prd/0001-event-sourced-orders.md` папка будет `materials/issues/0001-event-sourced-orders/`.
- `<NNN>` — 3-значный порядковый номер issue внутри папки, начиная с `001`.
- `<kebab-slug>` — короткий kebab-case-слаг, отражающий суть задачи.

Пример: `materials/issues/0001-event-sourced-orders/001-add-order-events-schema.md`.

Если папка `materials/issues/<...>/` ещё не существует — создай её. Тело файла заполни по шаблону ниже.

Создавай файлы в порядке зависимостей (сначала блокирующие), чтобы в поле «Блокируется» можно было сослаться на реальные имена уже созданных issue-файлов.

<issue-template>
## Родительский PRD

Ссылка на исходный PRD-файл в `materials/prd/` (относительный путь).

## Что строим

Краткое описание этой вертикальной задачи. Описывай end-to-end поведение, а не послойную реализацию.

Избегай конкретных путей к файлам и фрагментов кода — они быстро устаревают. Исключение: если прототип выдал фрагмент, который выражает решение точнее, чем проза (state machine, reducer, схема, форма типа), — вставь его сюда и кратко отметь, что он пришёл из прототипа. Оставь только те части, где принимаются решения, — не рабочее демо, а только важные фрагменты.

## Критерии выполненной задачи

- [ ] Критерий 1
- [ ] Критерий 2
- [ ] Критерий 3
- [ ] Тесты на <модуль> согласно «Решения по тестированию» родительского PRD *(добавляй этот пункт только если затрагиваемый задачей модуль перечислен в разделе «Решения по тестированию» PRD; иначе — опусти)*

## Блокируется

- Имя файла блокирующей issue (если есть), например `001-add-order-events-schema.md`

Или «Нет — можно начинать сразу», если блокеров нет.

</issue-template>

НЕ изменяй родительский PRD-файл.
