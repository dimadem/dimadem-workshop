# To PRD

Скилл берёт контекст текущего диалога и понимание кодовой базы и создаёт PRD-файл в `materials/prd/`. Этот файл далее используется скиллом [[to-issues]] для разбиения на задачи. НЕ устраивай интервью с пользователем — просто синтезируй то, что уже знаешь.

## Процесс

### 1. Изучи кодовую базу

Если ты ещё не изучил кодовую базу — сделай это, чтобы понять её текущее состояние. Используй словарь доменной области проекта на протяжении всего PRD и учитывай ADR (architectural decision records) в той части кода, которую затрагиваешь.

Также прочитай уже существующие PRD в `materials/prd/`, чтобы сверять терминологию и не противоречить ранее принятым решениям.

### 2. Спроектируй основные модули

Набросай основные модули, которые нужно будет построить или изменить для реализации. Активно ищи возможности выделить **глубокие модули**, которые можно тестировать в изоляции.

Глубокий модуль (в противоположность поверхностному) — это модуль, который инкапсулирует много функциональности за простым тестируемым интерфейсом, который редко меняется.

Сверься с пользователем, что предложенный набор модулей соответствует его ожиданиям. Уточни у пользователя, для каких модулей он хочет, чтобы были написаны тесты.

### 3. Напиши PRD-файл

Положи PRD в `materials/prd/` с именем вида `NNNN-kebab-case-name.md`, где:

- `NNNN` — следующий свободный 4-значный порядковый номер. Определи его, посмотрев на уже существующие файлы в `materials/prd/`: если последний `0003-...`, новый будет `0004-...`. Если папка пустая или ещё не существует — начни с `0001`.
- `kebab-case-name` — короткий kebab-case-слаг, отражающий суть фичи (например `0004-event-sourced-orders.md`).

Если папка `materials/prd/` ещё не существует — создай её.

Тело файла заполни по шаблону ниже.

<prd-template>

## Постановка проблемы

Проблема, с которой сталкивается пользователь, с точки зрения пользователя.

## Решение

Решение проблемы, с точки зрения пользователя.

## User Stories

ДЛИННЫЙ нумерованный список user stories. Каждая user story в формате:

1. Как <актор>, я хочу <фичу>, чтобы <выгода>

<user-story-example>
1. Как клиент мобильного банка, я хочу видеть баланс на своих счетах, чтобы принимать более обоснованные решения о своих тратах
</user-story-example>

Этот список user stories должен быть максимально подробным и покрывать все аспекты фичи.

## Решения по реализации

Список решений по реализации, которые были приняты. Сюда может входить:

- Модули, которые будут построены/изменены
- Интерфейсы этих модулей, которые будут изменены
- Технические уточнения от разработчика
- Архитектурные решения
- Изменения схемы
- API-контракты
- Конкретные взаимодействия

НЕ включай конкретные пути к файлам и фрагменты кода. Они быстро устаревают.

Исключение: если прототип выдал фрагмент, который выражает решение точнее, чем проза (state machine, reducer, схема, форма типа), — вставь его внутрь соответствующего решения и кратко отметь, что он пришёл из прототипа. Оставь только те части, где принимаются решения, — не рабочее демо, а только важные фрагменты.

## Решения по тестированию

Список решений по тестированию, которые были приняты. Включи:

- Описание того, что делает тест хорошим (тестировать только внешнее поведение, а не детали реализации)
- Какие модули будут покрыты тестами
- Примеры для ориентира (т.е. похожие типы тестов в кодовой базе)

## За рамками

Описание того, что находится за рамками этого PRD.

## Дополнительные заметки

Любые дополнительные заметки о фиче.
